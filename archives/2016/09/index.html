隔离级别RR & 存储引擎innodb

两个session测试GAP
(注意:对下述测试而产生锁名字分类{GAP或NextKey Lock}不做辨析,只分析产生锁的特征,且认为GAP)
GAP原理
当执行DML写操作如delete by key(key非唯一索引)时,产生GAP，区间为key前后查找临界值Key1 key2,形成锁间隙区间[key1,key2),注意右区间为开!!!;


若Key1或Key2无,则对应区间无界。


CREATE TABLE `tradeorder` (
`orderId` BIGINT(21) NOT NULL,
`buyerUserId` BIGINT(21) DEFAULT NULL,
`sellerUserId` BIGINT(21) DEFAULT NULL,
`shipTime` BIGINT(21) DEFAULT NULL,
KEY `orderId` (`orderId`)
) ENGINE=INNODB DEFAULT CHARSET=utf8
orderId非PK,非唯一索引

2ND TEST

Session1:
BEGIN;
START TRANSACTION;
DELETE FROM tradeorder WHERE orderid=2;
SELECT SLEEP(20);
COMMIT;
获取索引行为2的GAP,观察session2阻塞.


Session2:
INSERT INTO `tradeorder` (`orderId`, `buyerUserId`, `sellerUserId`, `shipTime`) VALUES('-100','-100','-100','-100'); /*阻塞*/
INSERT INTO `tradeorder` (`orderId`, `buyerUserId`, `sellerUserId`, `shipTime`) VALUES('1','5','2','2'); /*阻塞*/
INSERT INTO `tradeorder` (`orderId`, `buyerUserId`, `sellerUserId`, `shipTime`) VALUES('2','5','2','2'); /*阻塞*/
INSERT INTO `tradeorder` (`orderId`, `buyerUserId`, `sellerUserId`, `shipTime`) VALUES('3','5','2','2'); /*阻塞*/
INSERT INTO `tradeorder` (`orderId`, `buyerUserId`, `sellerUserId`, `shipTime`) VALUES('4','5','2','2'); /*阻塞*/
INSERT INTO `tradeorder` (`orderId`, `buyerUserId`, `sellerUserId`, `shipTime`) VALUES('200','200','200','200'); /*阻塞*/


测试结果:当表tradeorder empty时,所有insert均阻塞.
Insert key否收到GAP，取决于当前存在数据行.
如当前存在orderid 1、2、3,那么锁区间为[1,3),insert3不阻塞,insert1阻塞
当前存在orderid 1、2,那么锁区间为[1,∞),inset -100不阻塞,insert3阻塞,insert 100亦阻塞
