Consistency Mark 2ND

  • World

主题:可靠消息(事件)模式

一 分析

形态1
@Transactional
public void serviceMethod(){
   Boolean result=dao.pesist();
  if(result){mq.send();}
}
serviceMethod为事务边界,mq consumer 与 local dao 顺序无关性

形态2
public void serviceMethod(){
   try{
   Boolean result=dao.pesist();
        if(result){mq.send();}
  }
   catch(E e){
      rollBack();
   }
}
dao.pesist()方法为事务边界,mq consumer 与 local dao 顺序先后.

消费者端可选形态
public void consumer(final Object message) throws Exception{
  Boolean result=handle();
  if(!result){
       throw new RuntimeException();
  }
}

代码形态如上两种,本质上一致,Spring事务可see DataSourceTransactionManager 系列方法doBegin,doCommit==,随意选取形态2作下文分析.

执行情况枚举:
1.操作数据库dao.pesist()成功，mq.send()发送消息也成功.
2.操作数据库dao.pesist()失败,或者持久化异常，mq.send()不执行,不发送消息.
3.操作数据库dao.pesist()成功，但是发送消息事件mq.send()失败，throw异常，刚执行的数据库操作dao.pesist()被回滚。
基于上三种情况，感觉这个代码模式已经OK了，完美实现了 "A服务与B服务调用一致性,引入中间件M,A与M一致性，B与M亦一致性,AB最终一致".

危险时间枚举:
4 形态1中,当在mq.send()成功后,到操作数据库提交 com.mysql.jdbc.ConnectionImpl.commit() 之间，
如果服务"dao.pesist()"对应服务器或者DB宕机 也将造成数据库操作因为连接异常关闭而被回滚,最终结果是mq.send()发送消息成功，数据库却被回滚。

5.操作数据库dao.pesist()成功，mq.send()发送消息至Broker成功,但send()方法抛出异常,操作数据库dao.pesist()被回滚;此种场景应不会发生.
基于如RabbitMQ此种产品发送消息普遍无ACK，对生产者保证消息不丢失.

二 事件表

使用本地中间表M，存储消息(事件),形态如:
public void serviceMethod(){
Boolean result=dao.pesistBusinessAndMessage(); /*持久化业务与消息事件*/
  if(result){
        try{
           mq.send();
           Boolean result=dao.deleteMessage();
         }
        catch(E e){}
   }
}

执行情况枚举:
1 操作数据库dao.pesistBusinessAndMessage()成功，mq.send()发送消息也成功.
2.操作数据库dao.pesistBusinessAndMessage()失败,或者持久化异常，mq.send()不执行,不发送消息.
3 操作数据库dao.pesistBusinessAndMessage()成功，mq.send()发送消息失败,无所谓.

说明:
同一个本地事务中记录业务数据和消息事件数据,消息记为"待消费",正常情况mq.send()发送消息成功后 删除中间表"待消费"消息标示;
执行情况3,当发送消息失败,采取定时 Schedule Task方式，从中间表M恢复未发布成功的消息(事件),重新mq.send(),重新发送成功后再 删除中间表"待消费"消息标示;
使用本地中间表，将消息事件与业务耦合,增加额外的数据库操作，中间表可能成为瓶颈;
增强:将本地中间表,扩展为外部服务.

三 幂等性

问题枚举
1 消息的重复消费
2 消息的并发消费
3 消息消费顺序性

说明:MQ消费者实现幂等性与RPC手法基本还是一致
1 使用事件表M,根据业务可设计为状态机式state,如"未发送","已发送","已处理"，通过message全局Id,来获取消息(事件)的处理状态，发现重复消息.
全局Id也挺方便设计，可选使用业务类型维度与业务id维度等组合;
此种方法可以解决"消息的重复消费",但解决不了"消息的并发消费".

2 一个Q中存在两个相同的消息且投递给不同消费者线程并发处理,我能想到的是乐观锁.

3 当消费者端 接受消息m1,m2需要按顺序处理,若先处理m2后即m1不处理;使用版本号,针对某个消费者行为 全局递增序列,消费时判断版本号.

那么满足2、3即可实现幂等性,还好这些代码实现起来都没啥问题.

总结:最终一致性-可靠消息(事件)模式 的需要条件

1. MQ保证消息不丢,发送至Broker保证;
2. Consumer要保证自己接受处理完成后才发送ACK给Broker,中间件可保证
3. 每个Consumer做到对任何消息处理的幂等性

以上分析未使用代码测试,只做分析性伪代码, 编码精神乏力.
